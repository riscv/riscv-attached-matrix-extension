[[programming_model]]
== Programming Model
:imagesdir: ../images

=== Instruction stream

AME instructions are in-line with other instructions on the hart.
They are executed in program order, though follow a weaker memory model than RVWMO such that
AME memory operations not guaranteed to appear in program order relative to non-AME memory operations
without an ordering fence (see TODO).

=== State

Each hart implementing AME adds its own, private, AME registers and CSRs.
AME instructions execute on the hart to access that state and control data transfer between AME
registers and memory.

.Register state and data movement paths for AME. The F/D and V extensions are _optional_ with AME. The only architectural datapath between vector (if present) and AME is through memory. The scalar datapath(s) is/are one-way. AME accumulator state can only be moved to/from the AME matrix registers.
image::ame_state.svg[AME state and architectural datapaths]

The `Acc` registers are used to accumulate the output of partial matrix multiplies.
The `M` registers are general-purpose matrix registers.
They are used for anything that is not an accumulation, including the data operand of load or store instructions,
the input operands of 1D (element-wise/vector) operations, and the output operands of 1D operations.

A single `M` register may hold multiple, one, or part of a square,
depending on implementation-defined parameters and programmed datatype.
Generally, the number of squares that will fit in an `M` register is inversely proportional
to the datatype size (because larger datatypes require more storage).

In contrast, an `Acc` register always holds exactly one square.

=== Squares

All AME computational instructions operate on fixed-sized NxN square matrices.

[NOTE]
--
This constraint keeps the ISA and hardware simple: every compute instruction consumes and produces
the same logical shape (`N x N`), so instruction encoding does not need per-operand shape fields,
register mapping is uniform, and implementations can use fixed datapaths and scheduling.

For contrast, general matrix multiply on non-square inputs (`M x K` by `K x N`) produces `M x N`.
AME handles this by tiling into `N x N` squares and zero-padding boundary tiles in software.

.Example (`N = 8`)
* Multiply `A(3x5)` by `B(5x2)`.
* Software packs `A` and `B` into zero-padded `8x8` squares.
* AME computes a full `8x8` result square.
* Software keeps only the valid `3x2` upper-left region and discards padded rows/columns.
+
image::ame-square-padding-example.svg[Matrix padding example for N=8]

Larger non-square problems are handled the same way: process full interior `N x N` tiles normally,
and pad only edge tiles.
--

The row/column dimension, N, is implementation-defined and is equal to the
square root of <<udb:doc:param:AME_NELEM,AME_NELEM>>.
<<udb:doc:param:AME_NELEM,AME_NELEM>> can be any power of 4.

[NOTE]
--
It is possible to create portable AME binaries regardless of the chosen value of
<<udb:doc:param:AME_NELEM,AME_NELEM>>.

Highly-tuned binaries may produce a better result by targeting a specific value of
<<udb:doc:param:AME_NELEM,AME_NELEM>>, similar to how auto-vectorizing compilers can generate
more efficient code for RISC-V Vector when VLEN is known.
--

All elements within a single square have the same datatype, for example unsigned 8-bit integer or
single-precision IEEE floating point.

A single operation may reference squares with different datatypes (for example, 8-bit integer inputs widening to a 16-bit integer output).
Mixed-datatype operations are optional; see <<datatypes>> for the implementation discovery mechanism.

==== Mapping squares to M registers

A single square (vector or matrix) may fit in one `M` register or may span multiple consecutive
`M` registers, depending on the datatype size.
If the datatype is no wider than <<udb:doc:param:AME_UNIT_DATATYPE_SIZE,AME_UNIT_DATATYPE_SIZE>>,
one register is sufficient;
wider datatypes require additional registers.
The required register count is the datatype size rounded up in units of
<<udb:doc:param:AME_UNIT_DATATYPE_SIZE,AME_UNIT_DATATYPE_SIZE>>
(that is, `ceil(sizeof(datatype)/AME_UNIT_DATATYPE_SIZE)`).

When a square requires multiple consecutive `M` registers, consecutive registers are combined into a group.
When registers are grouped, the register index must be a multiple of the group size (_e.g._, when
group size is two, register indices must be even).
Specifying a register index that is not a multiple of the group size results in an
Illegal Instruction exception.
This mechanism is analogous to the `LMUL` facility used in the RISC-V vector (`V`) extension.

When the programmed datatype is a *storage-only datatype*
(size < `<<udb:doc:param:AME_UNIT_DATATYPE_SIZE,AME_UNIT_DATATYPE_SIZE>>`),
multiple logical squares are packed into a single M register.
The number of squares per register is the *pack factor*:
`pack_factor = AME_UNIT_DATATYPE_SIZE / sizeof(dtype)`.
Storage-only registers cannot be used as operands in compute instructions;
`<<udb:doc:inst:mconv_ew,mconv.ew>>` must first expand the data into
compute-dtype registers before computation.

.Square-to-M-register mapping examples
image::ame-square-reg-mapping.svg[Square to M register mapping examples]

When an `M` register is part of a group, only the lowest-indexed register of that group may be used directly in operations.
Using any other register in the group results in undefined behavior.

// not really bash, but it gets us colored # comments
[source,bash]
----
# precondition:
# a square is in m[2-3]

madd.1d m2, m2, m2  # well-defined, m2 = m2 + m2

madd.1d m2, m3, m3  # undefined
madd.1d m3, m2, m2  # undefined
----

=== Polymorphic instructions

AME instructions do not encode operand datatypes; instead, datatypes are determined by dedicated datatype registers.

[NOTE]
This feature makes AME analogous to polymorphic programming languages
– a single function (instruction) is defined for all datatype combinations (polymorphic argument types).

=== Datatypes

AME supports up to `2^32 - 1` unique datatypes, half of which are reserved for custom use.
The value 0 is reserved to mean uninitialized.

[NOTE]
--
The datatype registers are similar to WARL CSR fields, so implementations only need to support
enough state to represent the datatypes they support.

For example, an implementation that only supports two datatypes would only need two bits to
represent those datatypes plus uninitialized.
--

Standard datatypes and encoding rules are listed in <<datatypes>>.

Implementations may choose which standard datatypes are supported and which custom datatypes
are supported.

A legal implementation may support either zero standard datatypes or zero custom datatypes (i.e.,
all custom or all standard).

[NOTE]
--
Profiles will likely mandate a minimal set of supported datatypes.
--

Datatypes are set using the <<udb:doc:inst:msettyp,msettyp>> instruction and
read using the <<udb:doc:inst:mgettyp,mgettyp>> instruction.

==== Compute and storage datatypes

For a given implementation, datatypes can be partitioned into *compute* datatypes and *storage*
datatypes. Compute datatypes are those that can be used with AME 1D and/or 2D computation instructions.
Storage datatypes are those that can _only_ be used as the destination or source of a load or store.
Any datatype whose size is less than <<udb:doc:param:AME_UNIT_DATATYPE_SIZE,AME_UNIT_DATATYPE_SIZE>> _must_ be a storage datatype.
Storage datatypes are useful to handle packed datatype formats; see <<mapping-squares-to-m-registers>> for more details.

=== Data movement

Data movement between AME register files, the integer (X) register file, the floating point (F)
register file, and memory are restricted in order to facilitate a detached AME execution unit.

There are no AME data movement instructions between memory and the `Acc` registers;
rather, `Acc` registers can only be moved in to or out of the `M` registers.

`M` register data can be loaded and stored from/to memory.

A scalar from the X or F register file can be sent to an `M` register.

AME does not define data movement operations between AME register files and the vector (V)
register file.
Performance-sensitive AME code should use AME 1D operations on matrix data rather than RISC-V Vector.

=== Memory

AME uses the same address space as the host hart.
In systems with virtual memory, AME loads and stores go through (H)S-mode translation and,
if applicable, G-stage translation when run within a hypervisor.

AME provides memory storage variants that operate on one of three memory layouts:

*	Row-major (`mls.rm` / `mss.rm`)
*	Column-major (`mls.cm` / `mss.cm`)
*	Opaque, implementation-defined (`mls` / `mss`)

The row-major and column-major variants are called the _interoperability_ storage instructions.
They transfer data as a contiguous block — no striding or partial-tile predication is supported.

Behavior is only well-defined when loading an opaque tensor layout that was previously stored
opaquely on the same implementation.
Generally, software should assume that loads / stores using the opaque format will be more
performant than loads / stores using one of the formats with a standard layout.

==== Consistency model

Memory transfers from/to `M` registers follow a weaker memory model than other memory transfers on
the hart (from/to the X, F, or V register files).
Informally, memory transfers involving AME state may appear out of program-order with respect to
non-AME memory transfers in the hart.
Ordering is restored with QQQ (fence.ame instruction, ldaq, strl).

=== Operators

AME supports six broad classes of arithmetic and/or logical operators:

*	2D Matrix multiply
*	1D Element-wise/vector
*	Intra-tensor (reduction/expansion/permutation)
*	Register move / data conversion
*	Scalar broadcast
*	Load/store

AME does not support general masking/predication on computational instructions, though a future extension may add support.



==== 2-dimensional matrix multiply operations

Full N^3^ matrix multiplication of squares is a primitive operation of AME, performed using a single instruction.

AME matrix multiplication instructions calculate the matrix product, `C` of two squares, `A` and `B`.

Each output element `C[i,j]` is computed as the dot product of:

* row `i` of `A`, and
* column `j` of `B`.

The accumulation order for additions inside each dot product is implementation-defined.
That order is fixed within a given implementation, so repeated runs on the same implementation
produce the same result.

[NOTE]
Different implementations may use different accumulation orders, and therefore may produce
different results (especially for floating-point data).

Formally:

.AME matrix multiplication
[source]
--
def perm(range): implementation-defined permutation over range

for i from 0 to N-1:
  for j from 0 to N-1:
    C[i,j] = sum(A[i,k] * B[k,j] for k in perm(0..N-1))
--

Matrix multiplication is not commutative (`A*B != B*A` in general),
so operand order must be preserved by software using AME primitives.

AME defines [Udb::Instruction#mmul.2d, Udb::Instruction#mmulacc.2d, Udb::Instruction#mmulaccneg.2d, Udb::Instruction#mmulat.2d, Udb::Instruction#mmulatacc.2d, Udb::Instruction#mmulbt.2d, Udb::Instruction#mmulbtacc.2d, Udb::Instruction#mmulneg.2d] instructions that perform a matrix multiply.
The destination of matrix multiply operations is always an accumulator register.
The operations differ in
whether or not they accumulate into the destination register (versus overwrite it),
whether or not they transpose an input before multiplying,
and whether or not they negate a result.

==== 1-dimensional elementwise operations

AME defines 1-dimensional (elementwise/vector) operations that use the `M` registers.

Element-wise operations apply the same operator independently at each coordinate of the input squares.
For every position `(i,j)`, the instruction reads `A[i,j]` and `B[i,j]`, computes the result, and
writes it to `C[i,j]`.

There are four classes of elementwise operations: Arithmetic, Bitwise, Comparison, and Nonlinear.

| <<udb:doc:inst:mscatmax_row,mscatmax.row>>
| `D[i, B[i,j]] = max(D[i,B[i,j]], A[i,j])`
| Elementwise scatter-max
| <<udb:doc:inst:mshift_ew,mshift.ew>>
| `D[i,j] = A[i, j+imm]`
| Elementwise row shift by constant offset

==== Register move / data conversion

`<<udb:doc:inst:mmov_m_m,mmov.m.m>>` copies all elements from one `M` register to another.
`<<udb:doc:inst:mmov_a_m,mmov.a.m>>` copies all elements from an `Acc` register to an `M` register.
In both cases, the source and destination datatypes must match; if they differ, `amestatus.UN` is set and the destination is left unchanged.

`<<udb:doc:inst:mzero_2d,mzero.2d>>` writes zero to every element of an `Acc` register,
typically used to initialize an accumulator before a matrix multiply sequence.

[cols="1,2,3",options="header"]
|===
| Instruction | Operation | Description
| <<udb:doc:inst:mmov_a_m,mmov.a.m>>
| `D[i] = acc[i]`
| Elementwise unconditional move from accumulator to M register
| <<udb:doc:inst:mmov_m_m,mmov.m.m>>
| `D[i] = A[i]`
| Elementwise unconditional move between M registers
| <<udb:doc:inst:mzero_2d,mzero.2d>>
| `C = 0`
| Clear accumulator tile
|===

`<<udb:doc:inst:mconv_ew,mconv.ew>>` converts element data from one datatype to another.
An expand converts a single packed `M` register (storage-only dtype) into `pack_factor` consecutive
compute-dtype `M` registers; a compress does the reverse.

==== Scalars

In any AME implementation, software can use scalar broadcast instructions to use the same scalar
value with every element.
For example, software can implement elementwise add by a common constant using a scalar broadcast
followed by an elementwise add.

An optional extension, `Zvames`, provides instructions that take scalars as an input for elementwise
operations.
With `Zvames`, elementwise add by a common constant is a single instruction.

===== Scalar broadcast

AME provides two instructions to fill every element of an M register with a single scalar value,
analogous to `np.full_like`. The element size is determined by the register's programmed datatype.
Both compute and storage-only dtypes are supported; for storage-only dtypes all
`pack_factor * AME_NELEM` element positions are filled.

`<<udb:doc:inst:mbcast_x,mbcast.x>>` sources the scalar from an integer (`X`) register.
`<<udb:doc:inst:mbcast_f,mbcast.f>>` sources the scalar from a floating-point (`F`) register
and requires the F extension; software is responsible for placing the correctly-formatted
bit pattern for the destination datatype into the F register beforehand.

[cols="1,2,3",options="header"]
|===
| Instruction | Operation | Description
| <<udb:doc:inst:mbcast_x,mbcast.x>>
| `D[i] = xs1`
| Broadcast integer scalar to all elements of a square
|===

===== Scalar operand instructions

==== Load and store

The row-major and column-major interoperability instructions (`mls.rm`, `mls.cm`, `mss.rm`, `mss.cm`)
support both compute and storage-only datatypes. Data is always transferred as a single
contiguous block — no striding, partial-tile, or predicated variants exist.

For *compute datatypes* (size >= `AME_UNIT_DATATYPE_SIZE`), one square is transferred
between memory and one (or more, for wide types) M registers.

For *storage datatypes* (size < `AME_UNIT_DATATYPE_SIZE`), `pack_factor` squares are
transferred in a single instruction:

* **Load:** `pack_factor` squares are loaded from contiguous memory and packed into a single
  M register. In row-major format, sub-square `sq` occupies the `sq`-th block of
  `sqrt(AME_NELEM)` rows in memory. In column-major format, sub-square `sq` occupies the
  `sq`-th block of `sqrt(AME_NELEM)` columns.

* **Store:** The single packed M register is unpacked and `pack_factor` squares are written
  to contiguous memory using the same layout.

The typical workflow for storage-only data is:

// not really bash, but it gets us colored # comments
[source,bash]
----
  # Load INT4 data (pack_factor = AME_UNIT_DATATYPE_SIZE / 4 squares per register)
  li t0, INT4_DTYPE
  msettyp m0, t0
  mls.rm m0, a0                  # loads pack_factor INT4 squares into m0

  # Expand INT4 → INT16 before compute
  li t0, INT16_DTYPE
  msettyp m4, t0                # m4, m5, ... will hold the expanded squares
  mconv.ew m4, m0               # expands m0 into m4..m(4+pack_factor-1)

  # Compute using m4..m(4+pack_factor-1) as normal INT16 registers
  madd.1d m8, m4, m4

  # Optionally compress back to INT4 for storage
  li t0, INT4_DTYPE
  msettyp m0, t0
  mconv.ew m0, m8               # compresses m8..m(8+pack_factor-1) → m0
  mss.rm m0, a3                  # stores pack_factor INT4 squares from m0
----

The opaque-format instructions (`mls`, `mss`) transfer the raw M register contents without
interpreting the datatype and are correct for both compute and storage-only dtypes.
